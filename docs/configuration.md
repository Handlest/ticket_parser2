# Конфигурация (`setup.yaml`)

Все настраиваемые параметры приложения живут в одном файле `setup.yaml`.
Значения вида `${VAR}` подставляются из переменных окружения (файл `.env`).
Структура валидируется на старте через pydantic — при опечатке приложение упадёт
с понятной ошибкой.

## Полное описание полей

### `telegram`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `token` | строка | — (обязательно) | Токен бота. Обычно `"${TELEGRAM_BOT_TOKEN}"` — подставляется из `.env`. |
| `allowed_user_ids` | список чисел или строка | `[]` | Telegram `user_id`, которым разрешён доступ. Обычно `"${ALLOWED_USER_IDS}"` — подставляется из `.env` строкой через запятую (`"111,222"`). Можно задать и списком прямо в YAML. Пусто = доступ всем. |

### `database`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `path` | строка | `data/tracking.db` | Путь к файлу SQLite. Каталог создаётся автоматически. В Docker каталог `data/` смонтирован как volume. |

> Миграций нет: таблицы создаются на старте. Если поменяется схема — удали файл
> `.db` (пересоздастся) либо добавь колонку вручную. В БД две таблицы:
> `trackings` (отслеживания) и `airports` (кэш «город → IATA-код», заполняется
> автоматически; см. [providers.md](providers.md)).

### `scheduler`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `default_check_interval_minutes` | число | `60` | Рекомендуемый интервал, который бот предлагает при добавлении отслеживания. |
| `min_interval_minutes` | число | `15` | Минимально допустимый интервал проверки (защита от слишком частых запросов). Также задаёт частоту «тика» планировщика. |

### `providers`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `enabled` | список строк | `[]` | Имена активных провайдеров (`s7`, `redwing`, ...). См. [providers.md](providers.md). |
| `request_timeout_seconds` | число | `30` | Таймаут на один запрос / рендеринг страницы. |
| `headless` | bool | `true` | Запускать браузер (Playwright) в headless-режиме. На сервере — всегда `true`. |

### `locale`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `currency` | строка | `RUB` | Валюта (записывается в отслеживание, показывается в уведомлениях). |
| `language` | строка | `ru` | Язык интерфейса. |

### `logging`

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `level` | строка | `INFO` | Уровень логирования: `DEBUG`, `INFO`, `WARNING`, `ERROR`. |
| `file` | строка | `data/logs/app.log` | Файл со всеми логами уровня `level` и выше. Пустая строка = не писать в файл. |
| `error_file` | строка | `data/logs/errors.log` | Отдельный файл только для ошибок (`ERROR` и выше). Пустая строка = не писать. |
| `max_file_size_mb` | число | `5` | Максимальный размер файла лога перед ротацией. |
| `backup_count` | число | `3` | Сколько архивных копий хранить при ротации. |

Логи пишутся одновременно в консоль (видно в `docker compose logs -f`) и в файлы.
Каталог `data/logs/` лежит внутри volume `data/`, поэтому логи сохраняются между
перезапусками контейнера. Ошибки провайдеров (см. `BaseProvider.safe_search`)
попадают в `error_file` со стектрейсом.

### `search_log`

Лог результатов каждого похода на сайт: какие билеты были найдены и по какой
минимальной цене. Пишется отдельным логгером, чтобы не засорять общий лог.

| Поле | Тип | По умолчанию | Описание |
|------|-----|--------------|----------|
| `enabled` | bool | `false` | Вести ли лог поисков. `false` = ничего не пишем. |
| `file` | строка | `data/logs/searches.log` | Отдельный файл лога поисков. Пусто = писать в общий лог/консоль. |

Ротация переиспользует параметры `logging` (`max_file_size_mb` / `backup_count`,
по умолчанию 5 МБ и 3 копии). Запись на один провайдер и одно отслеживание:

```
2026-07-01 10:00:00 S7 Airlines | Москва→Сочи | 2026-07-15 | Без льготы | офферов: 3 | мин. цена: 5990 RUB
    - 5990 RUB · S7 1234 · 08:10–11:25 · Базовый
    - 7300 RUB · S7 1234 · 08:10–11:25 · Стандарт
    - 9100 RUB · S7 5678 · 14:05–17:20 · Базовый
```

## Пример

```yaml
telegram:
  token: "${TELEGRAM_BOT_TOKEN}"
  allowed_user_ids: "${ALLOWED_USER_IDS}"

database:
  path: "data/tracking.db"

scheduler:
  default_check_interval_minutes: 60
  min_interval_minutes: 15

providers:
  enabled: [s7, redwing]
  request_timeout_seconds: 30
  headless: true

locale:
  currency: "RUB"
  language: "ru"

search_log:
  enabled: true
  file: "data/logs/searches.log"

logging:
  level: "INFO"
```

## Применение изменений

В `docker-compose.yml` файл `setup.yaml` смонтирован внутрь контейнера, поэтому
после правки конфига достаточно перезапустить контейнер без пересборки образа:

```bash
docker compose restart
```
