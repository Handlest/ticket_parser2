# План проекта: бот отслеживания авиабилетов

## 1. Что делаем

Сервис, который по заданным пользователем параметрам периодически опрашивает
сайты авиакомпаний/агрегаторов, ищет билеты дешевле заданного порога (с учётом
льгот) и присылает уведомление в Telegram со ссылкой и ценой. Взаимодействие с
пользователем — только через Telegram-бота. Хранилище — SQLite. Всё запускается
на сервере одной командой через Docker.

---

## 2. Технологический стек

| Слой | Выбор | Краткое обоснование |
|------|-------|---------------------|
| Язык | **Python 3.12** | Требование заказчика; богатая экосистема парсинга. |
| Telegram-бот | **aiogram 3.x** | Современный async-фреймворк, FSM «из коробки» для пошагового добавления отслеживания. Заменил устаревший python-telegram-bot v13 как стандарт. |
| Парсинг (рендеринг JS) | **Playwright (async)** | Современная замена Selenium (прямо запрошено). Управляет реальным браузером, переживает JS-heavy сайты и базовый антибот, авто-ожидания элементов. |
| Парсинг (лёгкие/API провайдеры) | **httpx + selectolax** | Для провайдеров с открытым JSON-API или простым HTML браузер не нужен — это быстрее и дешевле по ресурсам. selectolax (на lxml) быстрее BeautifulSoup. |
| Планировщик | **APScheduler (AsyncIO)** | Интервальные задачи на пользователя; интеграция с asyncio, не нужен отдельный воркер/cron. |
| БД / ORM | **SQLite + SQLAlchemy 2.0 (async, aiosqlite)** | SQLite — простое требование заказчика. ORM-модели повышают читаемость и упрощают миграции схемы. |
| Конфиг | **PyYAML + pydantic-settings** | Все параметры в `setup.yaml` (требование), pydantic валидирует структуру при старте и даёт понятные ошибки. |
| Контейнеризация | **Docker + docker-compose** | Запуск одной командой `docker compose up -d`, образ Playwright уже содержит браузеры. |

> Примечание по реальности парсинга: сайты авиакомпаний активно защищаются от
> ботов. Архитектура провайдеров спроектирована так, чтобы провайдер мог быть
> как «лёгким» (httpx к внутреннему API), так и «браузерным» (Playwright).
> В поставке будут шаблонные провайдеры с понятной структурой; конкретные
> CSS-селекторы/эндпоинты придётся донастраивать под живой сайт — это вынесено
> в отдельные модули провайдеров, чтобы менять их без правки ядра.

## 3. Архитектура и компоненты

Ожидаемое количество модулей-компонентов: **~8 пакетов**. Принцип — ядро ничего
не знает о конкретных сайтах, провайдеры подключаются через реестр (registry).

```
plane_ticket_bot/
├── setup.yaml                  # ВСЕ настройки и параметры
├── Dockerfile
├── docker-compose.yml
├── requirements.txt
├── docs/                       # документация (ведётся по ходу проекта)
└── app/
    ├── main.py                 # точка входа: поднимает бот + планировщик
    ├── config/                 # (1) загрузка и валидация setup.yaml (pydantic)
    ├── db/                     # (2) модели SQLAlchemy + репозиторий + сессия
    ├── providers/              # (3) парсеры сайтов
    │   ├── base.py             #     BaseProvider (абстрактный интерфейс)
    │   ├── registry.py         #     реестр: имя -> класс провайдера
    │   ├── models.py           #     FlightOffer, SearchQuery (нормализованные)
    │   ├── s7.py               #     пример провайдера
    │   └── redwing.py          #     пример провайдера
    ├── core/                   # (4) логика отслеживания + матчинг по цене/льготам
    │   └── tracker.py          #     планирование проверок, очистка прошедших дат
    ├── bot/                    # (5) aiogram: хендлеры, FSM добавления, клавиатуры
    ├── notifications/          # (6) отправка уведомлений в Telegram
    └── domain/                 # (7) общие enum'ы: льготы, статусы
        └── benefits.py
```

(8-й компонент — `docs/` как живая документация.)

### Ключевые абстракции

- **`BaseProvider`** — абстрактный класс с методом
  `async search(query: SearchQuery) -> list[FlightOffer]`. Добавить S7 / RedWing /
  новый сайт = создать файл в `providers/`, унаследоваться, зарегистрировать в
  реестре. Ядро и бот не меняются. **Модульность — приоритет.**
- **`SearchQuery`** — нормализованный запрос (откуда, куда, дата, льгота…).
- **`FlightOffer`** — нормализованный результат (цена, ссылка, рейс, провайдер).
- **`Tracking`** (модель БД) — сохранённое отслеживание пользователя.

### Поток работы

1. Пользователь в боте: `/add` → FSM спрашивает город отправления, город
   назначения, дату, порог цены, льготу, интервал проверки → запись в SQLite.
2. APScheduler по интервалу запускает проверку: для каждого активного
   отслеживания опрашивает включённые провайдеры → собирает офферы.
3. Если найден оффер дешевле порога — уведомление в Telegram (ссылка + цена).
4. Перед/во время проверки: если дата вылета прошла — отслеживание удаляется.

## 4. Льготы на авиабилеты (исследование, попадёт в docs)

В РФ это в основном **программа субсидированных авиаперевозок** (продлена на 2026).
Категории, имеющие право на льготный тариф:

- **Молодёжь** — граждане до 23 лет.
- **Пенсионеры** — мужчины от 60 лет, женщины от 55 лет.
- **Многодетные семьи** — при наличии удостоверения.
- **Жители ДФО** — зарегистрированные в Дальневосточном федеральном округе.
- **Инвалиды** I группы, инвалиды с детства II/III групп и сопровождающие.

Перевозчики: в программе 2026 участвуют ~22 авиакомпании (Аэрофлот, S7, «Аврора»
— базовый перевозчик ДФО, «Россия», NordStar, «Якутия» и др.). Онлайн льготное
бронирование — у Аэрофлота и частично S7; у большинства — только в офисах.

В модели данных льгота — это enum (`BenefitCategory`), который передаётся
провайдеру как часть `SearchQuery`. Провайдер сам решает, поддерживает ли он
данную льготу. Полная таблица льгот/перевозчиков будет в `docs/benefits.md`.

## 5. Конфигурация (`setup.yaml`)

Всё настраиваемое — в одном файле. Черновик структуры:

```yaml
telegram:
  token: "${TELEGRAM_BOT_TOKEN}"   # токен подставляется из .env (создаёшь ты)
  allowed_user_ids: []             # пусто = доступ всем

database:
  path: "data/tracking.db"

scheduler:
  default_check_interval_minutes: 60
  min_interval_minutes: 15         # защита от слишком частых запросов

providers:
  enabled: ["s7", "redwing"]
  request_timeout_seconds: 30
  headless: true

logging:
  level: "INFO"
```
